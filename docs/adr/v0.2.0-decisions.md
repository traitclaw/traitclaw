# Architecture Decision Records — v0.2.0

## ADR-1: AgentStrategy Dispatch Mechanism

### Context

v0.1.0 hardcodes the agent loop inside `runtime::run_agent()`. Users cannot
customize the reasoning architecture (ReAct, MCTS, Chain-of-Thought) without
forking the crate.

### Decision

Use **dynamic dispatch** via `Box<dyn AgentStrategy>`.

### Rationale

- LLM call latency (200–2000ms) dwarfs vtable overhead (nanoseconds)
- Enables runtime strategy swapping without recompilation
- Simpler API: no generic parameter pollution on `Agent<S>`
- Object-safe trait works with `dyn` for heterogeneous agent collections

### Consequences

- Small heap allocation for the strategy box (one per agent, negligible)
- Strategy cannot be generic over associated types (acceptable trade-off)

## ADR-2: AgentHook Async Design

### Context

Hooks need to perform I/O (send metrics, log to services) without blocking
the agent loop. The existing `Tracker` trait is synchronous and serves the
internal Steering subsystem.

### Decision

Use **async methods via `async_trait`** (compatible with Rust 1.75+).
Hooks are separate from Tracker.

### Rationale

- Non-blocking I/O for observability services (DataDog, Prometheus)
- Clear separation: Tracker = internal state, Hook = external observability
- `before_tool_execute` returns `HookAction::Block` for security interception
- Multiple hooks via `Vec<Arc<dyn AgentHook>>` — composable

### Consequences

- Hooks add latency proportional to their async operations
- Sequential evaluation (not parallel) — intentional for deterministic ordering
- Blanket `Arc<T>` impl enables shared hook instances across agents

## ADR-3: Router Design for Multi-Agent Teams

### Context

`traitclaw-team` needs pluggable routing without forcing a graph dependency.

### Decision

Simple **`trait Router`** with `fn route(&self, message, state) -> RoutingDecision`.
No graph library dependency.

### Rationale

- Most routing patterns (sequential, leader-follower) are simple state machines
- `RoutingDecision` enum (`SendTo`, `Broadcast`, `Complete`) covers all patterns
- Graph-based routing can be built on top without core dependency
- `TeamState` provides enough context for complex routers (history, iteration)

### Consequences

- Complex graph topologies require custom Router implementations
- No built-in cycle detection (router author's responsibility)
- Default `SequentialRouter` preserves v0.1.0 behavior

## ADR-4: CompressedMemory Decorator Pattern

### Context

Long-running agents exceed context windows. Need automatic context
summarization that works with any `Memory` backend.

### Decision

**Decorator pattern**: `CompressedMemory<M: Memory>` wraps any `M` and
implements `Memory` itself.

### Rationale

- Works with `InMemoryMemory`, `SqliteMemory`, or any future backend
- Stackable: `CompressedMemory<CompressedMemory<SqliteMemory>>` compiles
- No changes to existing Memory implementations
- Threshold + keep_recent is simple to configure and understand

### Consequences

- Built-in summarization is simple concatenation (not LLM-powered)
- For production LLM summarization, users wrap with a custom strategy
- Summary cache uses `Mutex<HashMap>` — adequate for single-agent use
